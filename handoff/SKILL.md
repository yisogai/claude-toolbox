# handoff — コンパクト用の引き継ぎ生成

## これは何か

ユーザーは context を使い切る前に、手動で `/compact` して文脈を圧縮する運用をしている。`/compact [指示]` の引数は「**要約器に何を最優先で保持・破棄させるか**」の指示として効く。ただし**要約器はツールを持たずファイルを読めない**ため、引数にパスだけ書いても中身は要約に反映されない。そこでこのスキルは次のハイブリッド方式を取る:

1. 引き継ぎ全文を**ファイルに保存**する（`~/.claude/handoffs/`）。
2. `/compact` の引数には全文ではなく、「保存先の絶対パス＋**要約にこのパスと〈再開時はまずこのファイルを Read せよ〉を逐語で残せ**という要約器への指示」の短い一行だけを渡す。
3. compact 後の Claude は通常どおりツールを使えるので、「続けて」と言われた時点で要約に残ったパスを Read し、**無劣化の引き継ぎ全文**を取り戻して再開する。
4. さらに保険として、`SessionStart` hook（matcher: compact、`handoff_compact_hook.sh`、`~/.claude/settings.json` に配線済み）が compact 直後に引き継ぎ全文を**自動注入**する。要約器がパスを落としても復元が保証され、手動 handoff 後に**自動コンパクト**へ落ちた場合もカバーされる。

全文を引数に貼る旧方式より、(a) 貼り付けが短く事故らない、(b) 引き継ぎ内容が要約で劣化しない、(c) compact が失敗しても控えが残る、の3点で優る。

このスキルの役割は、**会話状態を解析して引き継ぎを生成 → ファイルへ保存 → 実行用の `/compact` コマンド一行をクリップボードへ入れ、同じコマンドをチャットにも表示する**こと。`/compact` の実行自体はユーザーが手動で行う（スキルから組み込みコマンドは起動できない）。

## まず判断: 生成するに値するか

会話がまだ数往復しかなく、保持すべき決定・現在地・次の一手がほとんど無いなら、**引き継ぎを生成せず**「まだ引き継ぎを作るほどの状態が溜まっていない」旨を伝えて終了する。中身の薄い引き継ぎを compact に渡すと、かえってノイズになる。

## 手順

1. **会話状態を解析する。** 直近だけでなくセッション全体の目的・確定した決定・現在地・次の一手を拾う。トランスクリプトの逐語再現ではなく、「決定」と「状態」を残す。再開する自分が迷わないことが基準。
2. 必要なら作業コンテキストも拾う（現在のブランチ、未コミット変更、編集中ファイル等）。関係するものだけ含め、無関係なら省く。
3. 下の**テンプレート**に沿って引き継ぎ本文を組み立てる。**運用規律**を守る。
4. 本文を **Write tool** で一時ファイル（例: `/tmp/claude-handoff-draft.md`）へそのまま書き出す。テンプレを囲う ``` フェンスは提示用なので**本文には含めない**（素のテキストにする）。本文の1行目は必ず冒頭の `[引き継ぎの扱い] …` から始める。
5. 同梱スクリプトに**その一時ファイルを STDIN で渡して**実行する。本文をシェル引数に乗せないことで、多バイト/記号/フェンスの破壊を避ける:
   ```
   bash "<このスキルのディレクトリ>/scripts/handoff_save.sh" < /tmp/claude-handoff-draft.md
   ```
   - 「このスキルのディレクトリ」はスキル起動時に表示される base ディレクトリ。ユーザーレベル導入なら通常 `~/.claude/skills/handoff/scripts/handoff_save.sh`。
   - ユーザーが「ファイルに残さないで」「保存不要」と言った場合のみ `--no-save` を付ける: `bash ".../handoff_save.sh" --no-save < /tmp/claude-handoff-draft.md`。この場合ファイル参照方式は使えないため、スクリプトは本文全文をクリップボードに入れる旧方式（MODE=body）にフォールバックする。
6. スクリプトの出力を読み、**報告フォーマット**に従って分岐して報告する。保存成功時（MODE=cmd）は最終行の前に `CMD: /compact …` という行があり、これが**ユーザーに実行してもらうコマンドの全文**（クリップボードにコピー済みのものと同一）。最終行は `OK=1 SAVE=… PATH=… CHARS=… CLIPBOARD=… OVERLONG=… MODE=…`。スクリプトの実測値（特に `CLIPBOARD` と `MODE`）に基づくこと——「コピーした」と決め打ちしない。

## 引き継ぎテンプレート

生成する本文はこの構造にする。**枠の ``` は提示用の囲みであり、ファイルへ書き出す本文には含めない。** 本文1行目は「このファイルを読んだ側」への指示にする。通常は compact 後の自分が Read で読むが、フォールバック（--no-save / 保存失敗）では本文がそのまま `/compact` 引数として要約器に渡るため、どちらが読んでも機能する文面にしてある。

```
[引き継ぎの扱い] これは手動コンパクト用の引き継ぎ。読んだ側はこの内容を最新の正として作業を再開すること（会話要約と食い違う場合はこちらを優先）。これを要約する場合は、決定事項・ファイルパス・設定値・数値・バージョン・コマンドを逐語で保持し、特に「今すぐの一手」「決定」「守るべき制約・嗜好」を欠落させないこと。

# 引き継ぎ: <タスクを一言で> (<YYYY-MM-DD>)

## 今すぐの一手（再開トリガー）
<compact後に「続けて」で即動ける最初の1手を1〜2行で。対象ファイル・コマンド・操作を具体に。曖昧語禁止>

## ゴール
- 最上位: <このセッションの最終目的。1〜2行>
- 現在のサブゴール: <今まさに取り組んでいる小目的。1行>

## 現在地
<どこまで進んだか。動いているもの／壊れているもの／未確認のもの。各1行。推測は(推測)と明記>

## 決定（理由つき・再議論しない）
- <採用した方針> ← 理由: <なぜ>
- <前提・制約として確定したこと>

## 検討して却下した案（理由つき）
- <却下した代替案> ← 却下理由: <なぜ採らないか>（再提案しないこと）

## 重要なファイル・場所
- `/abs/path/to/file.ext:行` — <役割・今の状態（編集中/未着手 等）>  ← 再開時に cwd が不明でも動くよう絶対パスで

## 検証・確認方法
- <テスト/起動/確認コマンド> → 期待される結果: <何が出れば成功か>

## 次のステップ
1. <具体的・順序付きアクション>
2. ...

## 守るべき制約・嗜好
- <ユーザーの嗜好（口調・粒度・言語）、触ってはいけない箇所、使ってはいけない手段 など>

## 注意・未解決
- <ハマりどころ、未解決の論点、避けるべき道>
```

## 運用規律

- 各項目は **1〜2行**。既知の事実のみを書き、推測には `(推測)` を付す。トランスクリプトの逐語転記は禁止。コード断片の全文貼り付けはせず、ファイルは `path:行` と役割で示す。
- 中身が無いセクションは**見出しごと落とす**。ただし「今すぐの一手」「ゴール」「次のステップ」は原則必須。
- 「今すぐの一手」を冒頭に置くのは、読む側（compact 後の自分／フォールバック時は要約器）に最重要の再開トリガーを最初に確実に目へ入れるため。

## 報告フォーマット

スクリプト出力の `MODE` でまず分岐し、次に `CLIPBOARD` で分岐する。引き継ぎ本文の全文をチャットに再掲する必要は**基本ない**（ファイルにあるため）。

**保存先（PATH）の表記ルール（全ブランチ共通）:** `SAVE=ok` のときだけ「引き継ぎ本文: <PATH>」と出す。`SAVE=skipped` なら「（ファイルは残していません）」、`SAVE=failed` なら「（ファイル保存に失敗したため、全文貼り付け方式にフォールバックしました）」に置き換える。`PATH=none` のときに「引き継ぎ本文: none」とは絶対に書かない。

### MODE=cmd（通常: ファイル保存に成功）

**必ず `CMD:` 行のコマンド全文を一字一句そのままコードブロックで表示する**（クリップボードに入っていても表示する。コピー失敗・文字化けの保険と、ユーザーが実行前に内容を確認できるようにするため）。

**`CLIPBOARD=ok` のとき:**
```
引き継ぎを保存しました（本文 <CHARS>字）。引き継ぎ本文: <PATH>

実行するコマンド（クリップボードにコピー済み・貼り付けるだけ）:

<CMD 行のコマンド全文をコードブロックで>

次の操作:
1. 入力欄に貼り付け（macOS: Cmd+V / Windows・Linux: Ctrl+V）→ Enter
2. compact 完了後「続けて」（continue 等でも可）→ 引き継ぎは hook が自動注入します（hook 未配線の環境でも要約に残ったパスから Read で復元）
```

**`CLIPBOARD=osc52` のとき**（リモート: tmux/OSC 52 で手元ターミナルへ転送。届いたかはスクリプトからは検証できない）: 上と同じ構成で、1行目の末尾を「実行するコマンドを OSC 52 で手元端末のクリップボードへ転送しました」にし、次の操作に「※貼り付けが空なら、お使いのターミナルが OSC 52 非対応です（VS Code 統合ターミナル / iTerm2 / Windows Terminal は対応、macOS 標準の Terminal.app は非対応）。その場合は上のコマンドを手でコピーしてください。」を添える。

**`CLIPBOARD=none` のとき:**
```
クリップボードに入れられませんでした。下のコマンドをコピーして、そのまま実行してください。引き継ぎ本文: <PATH>

<CMD 行のコマンド全文をコードブロックで>

compact 完了後は「続けて」（continue 等でも可）で再開します。
```

### MODE=body（フォールバック: --no-save または SAVE=failed）

クリップボードに入っているのは引き継ぎ本文そのもの。ユーザーは `/compact ` と入力してから貼り付ける（旧方式）。

**`CLIPBOARD=ok` のとき:**
```
引き継ぎをクリップボードにコピーしました（<CHARS>字）。（ファイルは残していません）

次の操作:
1. `/compact ` と入力（半角スペースを1つ）→ 貼り付け（macOS: Cmd+V / Windows・Linux: Ctrl+V）→ Enter
2. 完了したら「続けて」（continue 等でも可）で再開
```

**`CLIPBOARD=osc52` のとき:** 上と同じ構成で1行目を「引き継ぎを OSC 52 で手元端末のクリップボードへ転送しました（<CHARS>字）」にし、「※貼り付けが空なら、お使いのターミナルが OSC 52 非対応です（VS Code 統合ターミナル / iTerm2 / Windows Terminal は対応、macOS 標準の Terminal.app は非対応）。『全文出して』と言ってもらえれば本文をチャットに出します。」を添える。

**`CLIPBOARD=none` のとき**（コピー手段が無い/失敗）: 範囲ミスを避けるため、ここでは**全文をフェンス付きでチャットに出す**。保存先表記は上の共通ルールに従う。
```
クリップボードに入れられませんでした。下の全文を `/compact ` の後ろに貼ってください。

<引き継ぎ本文の全文をコードブロックで>
```

**`OVERLONG=1` のとき（MODE=body でのみ発生）:** 上記に加えて「長すぎて貼り付け/引数の上限を超える恐れがあります。コード全文を削る・ファイルは path:行のみにする等で短くして作り直しますか？」と警告する。

## 同梱リソース

- `scripts/handoff_save.sh` — 本文を STDIN で受け取り、(1) `~/.claude/handoffs/handoff-<日時>.md` へ一意な名前で保存（同一秒の衝突は連番で回避）、(2) 保存成功時はそのパスを参照する実行用 `/compact` コマンド一行を組み立てて `CMD: ` 行として出力し、**コマンドの方**をクリップボードへコピー（`MODE=cmd`。末尾改行なし＝貼り付けで即送信されない）。あわせて自動注入用マーカー `~/.claude/handoffs/.pending-<セッションID>`（Bash 環境の `CLAUDE_CODE_SESSION_ID`。無ければ無印 `.pending`）に保存先パスを書き、1日より古いマーカーは掃除する。保存が無いとき（`--no-save`／保存失敗）は従来どおり**本文全文**をコピーし（`MODE=body`）、以前の保存で残った自セッションのマーカーを無効化する（古い引き継ぎの誤注入防止）。コピー手段は環境を自動検出（非 macOS で tmux 内なら `tmux load-buffer -w` による OSC 52 転送を最優先 → `CLIPBOARD=osc52`。次いで pbcopy / wl-copy / xclip / xsel / WSL は文字化け回避で iconv→clip.exe）、(3) 結果を `KEY=VALUE` の最終行で出力（`MODE` キーを含む）。手段が無ければ `CLIPBOARD=none` を返す。
- `scripts/handoff_compact_hook.sh` — `SessionStart` hook（matcher: compact）。compact 直後にマーカーを読み、**保存時刻を明記した**引き継ぎ全文（24KB 超ならパス参照の指示のみ）を stdout 経由で新コンテキストへ自動注入し、マーカーを消す（consume-once）。注入文は「保存時点まではこの内容が正。保存より後の決定・進捗が要約にあればそちらを優先」と時刻で優先関係を条件付ける（auto-compact が数時間後に発火しても新しい決定を巻き戻さない）。自セッションのマーカー（12時間以内）を最優先、無印 `.pending` は他セッション由来の可能性があるため 60 分以内のみ採用。他セッションの `.pending-<別ID>` には触れない（並行セッションへの誤注入防止）。失敗時は必ず exit 0。`~/.claude/settings.json` に配線済み:
  ```json
  "SessionStart": [{ "matcher": "compact", "hooks": [{ "type": "command", "command": "bash ~/.claude/skills/handoff/scripts/handoff_compact_hook.sh" }] }]
  ```
- `scripts/context_usage.sh` — statusline / hook 共通ヘルパー。stdin の JSON から context 使用率（整数%）を算出。優先順位: (1) `.context_window.used_percentage`（正値）、(2) statusline が書くセッション毎キャッシュ `${TMPDIR:-/tmp}/claude-context-pct-<session_id>`（10分以内）、(3) transcript の最終 assistant usage からの概算（分母は `CLAUDE_CONTEXT_WINDOW`、既定 200000。100%超は分母想定違いとみなし 0 を返す）。`--strict` 指定時は (3) を使わない — threshold hook はこれで呼び、1M window モデルでの誤警告を原理的に防ぐ。
- `scripts/handoff_statusline.sh` — context 使用率を色付きで常時表示する statusline（<75% 緑● / 75–84% 黄⚠ / ≥85% 赤⚠）＋モデル名＋cwd。`~/.claude/settings.json` に配線済み:
  ```json
  "statusLine": { "type": "command", "command": "bash ~/.claude/skills/handoff/scripts/handoff_statusline.sh" }
  ```
- `scripts/handoff_threshold_hook.sh` — UserPromptSubmit hook。使用率 75/85/95% を超えた段階でセッション毎1回だけ、handoff を提案する additionalContext を注入する。失敗時は必ず exit 0（プロンプトをブロックしない）。`~/.claude/settings.json` の `hooks.UserPromptSubmit` に配線済み。

## インストール（共有して使う場合）

- **全プロジェクトで使う（推奨）:** この `handoff` フォルダごと `~/.claude/skills/handoff/`（中に `SKILL.md` と `scripts/` がある状態）に置く。
- **特定リポジトリだけで使う:** `<リポジトリ>/.claude/skills/handoff/` に置く。
- フォルダ名は `handoff`、メインファイルは `SKILL.md` で固定。置いたら Claude Code を起動し直し、入力欄で `/handoff` が候補に出ればロード成功。出なくても「ハンドオフ作って」と話しかければ起動する。
- クリップボードへのコピーは OS により使うコマンドが異なる（macOS: `pbcopy` / Windows: `clip`・PowerShell `Set-Clipboard` / Linux: `wl-copy`・`xclip`・`xsel`）。スクリプトが自動検出する。どれも無い環境では自動でチャットに全文を出すので、そこから手動コピーする。
- **SSH で入るリモートマシン（サーバー等）で使う場合**: リモート側のクリップボードに入れても手元では貼れないため、tmux 内なら `tmux load-buffer -w`（OSC 52）で**手元ターミナルのクリップボード**へ転送する。tmux 側は `set-clipboard`（既定 external で可）と、外側ターミナルを clipboard 対応とみなす `set -as terminal-features ',*:clipboard'` を `~/.tmux.conf` に入れておくと確実。
- 前提: POSIX sh（bash）で動く。素の Windows（cmd/PowerShell 単体、Git Bash も WSL も無い）は非対象。WSL か Git Bash を想定。

## プライバシーと後始末

- 引き継ぎは会話の要約なので、**API キーの断片・未公開コード・社内情報・固有名詞など機微情報を含む場合がある**。`~/.claude/handoffs/` に**平文で保存**され、自動削除はされない。（自動注入用の `.pending*` マーカーは保存先パスのみを含み、hook が消費するか 1 日で自動掃除される。）
- 機微な会話でファイルを残したくない場合は「ファイルに残さないで」と伝えれば、スクリプトを `--no-save` で呼びファイルを残さない。この場合ファイル参照方式は使えないため、本文全文をクリップボードへ入れて `/compact ` の引数に貼る旧方式（MODE=body）になる。
- 古い控えの整理（30日より古いものを削除）:
  ```
  find ~/.claude/handoffs -name 'handoff-*.md' -type f -mtime +30 -delete
  ```
  全消去は `rm -f ~/.claude/handoffs/handoff-*.md`。
- `~/.claude` を dotfiles リポジトリ等で git 管理しているなら、`.gitignore` に `handoffs/` を追加して誤コミットを防ぐ。

## 設計メモ

- 引き継ぎは「次の自分への手紙」。網羅より、再開時に迷わないことを優先する。決定の**理由**と**却下した案**を残すと、コンパクト後に同じ議論を蒸し返さずに済む。
- `/compact` 引数に全文でなく**ファイル参照つきの短いコマンド**を渡すのは、compact の要約器がツールを持たずファイルを読めない一方、**compact 後の Claude は Read できる**ため。要約には「パス＋まず Read せよ」が逐語で残れば十分で、引き継ぎ本体はファイルから無劣化で復元される（要約器がファイルを読めない点は公式ドキュメントに明記は無いが、要約は単発の LLM リクエストでツール実行の記載も無い。2026-07 時点の code.claude.com/docs/en/context-window.md 等で確認）。
- 復元は多層防御: (1) hook による自動注入（最も確実・自動コンパクトもカバー）、(2) 要約に残るパス＋Read（hook 未配線の環境用）、(3) ユーザーが控えのパスを自分で示す（最後の手段）。hook のマーカーを consume-once かつセッション紐付けにするのは、並行セッションの compact に別セッションの引き継ぎが注入される事故を防ぐため。session_id は compact 前後で不変（公式ドキュメント確認済み）。
- 本文をシェルに通さず Write→STDIN で渡すのは、多バイト・改行・バッククォート・``` の破壊を原理的に防ぐため。OS 差・命名・コピー手段といった決定論的処理はスクリプトに寄せ、モデルは「会話を解析して良い引き継ぎを書く」判断に集中する。
- ファイルを毎回残すのは、compact が失敗・中断したときの保険と、後から経緯を grep できる履歴のため。
